システムの目的

dramaturgy は、巨大で全体像のつかめない既存システムに初めて向き合う開発者や分析担当のためのツールで、ソースコードから「人間が短時間で読み解ける意味地図」を作り出すことを目的とする。狙いは、どんな業務領域があり、誰（登場人物）が何のために関与し、どんな中心概念（データ）を扱うのかを、必要に応じてコード・DB・APIへ降りられる形で一望できるようにすることにある。

全体は二者の協業で動く。確実な事実集め・整合検査・HTML描画といった機械作業はPython製CLIが担い、領域の切り出し・概念の圧縮・業務フローの意味づけといった判断は、ヘッドレス起動した Claude Code に任せる。操作者はローカルWeb UI（dra serve）から、リポジトリ目録の作成→領域ツリーの構築→領域ごとのカード生成→統合・検査→全体目的の付与→HTML描画という一連を一括で起動し、結果を眺める。

中心概念は領域ツリーと、領域カードを統合した正本「意味地図(meaning-map)」で、業務領域・登場人物・概念データ・CRUD・タグがそこに集約される。生成後も操作者は各項目へ三種類の指摘（再整理・検査・将来提案）を付けて地図を継続的に磨ける。UI言語と生成内容言語を分離した多言語設計により、対象システムに合わせた読みやすい地図を提供する。

アクター扱いのシステム

Claude Code

意味判断を担う生成主体。ヘッドレス起動され、目録とソースを読んで自然な業務領域ツリーを作り、必要な領域だけを子領域に細分化して area-tree.json を直接書き込む。

  • 領域ツリーを生成・細分化する (領域ツリーの構築) — 目録を手がかりにソースを読み、業務的に自然な領域ツリーを生成。子領域レビューで大きすぎる/責務混在の領域だけ分割する。
  • 領域カードを生成する (領域カード生成) — 列挙ファイルを読み、概念・CRUD・登場人物・フローを整理した area-maps/<id>.json を生成する。
  • 領域カードを生成する (意味地図の組み立てと検証) — 領域ツリーの各領域について area-maps/<id>.json を生成し、統合の入力を作る。
  • 指摘を処理する (対話的レビュー（指摘）) — reframe は正本を直接編集、audit は調査結果をJSONで、proposal は将来提案をMarkdownで記録する。
  • 概念にタグを付与する (タグ語彙（システム固有）) — カード生成時、語彙（グループ・意味込み）を参考に master/transaction 等のシステム固有タグを概念に付ける。
  • ヘッドレスで実行される (Claude Code 連携とジョブ実行) — stream-json で起動され、進捗イベントを流しつつ正本JSONを書き、session_id を返して --resume の継続を可能にする。
  • ヘッドレスで意味判断を実行する (Web UI とサーバ) — UIが起動したジョブとして領域ツリー/カード生成・指摘処理を実行し、正本JSONを更新する。
  • パイプラインの意味判断を実行する (HTTPサーバとAPI) — _run_init から起動され、領域ツリー/子領域/カード/システム全体目的を生成して正本JSONを更新する。

概念の区分: 指摘(finding)

指摘の種別
指摘の3種類。処理内容と反映先（正本を編集するか別ファイルに記録するか）が異なる。
reframe 再整理（正本 meaning-map.json を直接編集）audit 検査（正本不変・audits/<id>.json に記録）proposal 将来提案（as-is不変・proposals/<id>.md に記録）
指摘の状態
指摘のライフサイクル状態。open から running を経て done/error へ。再起動時 running は open へ戻される。dismissed は破棄。
open 待機中（キュー）running 実行中done 完了error エラーdismissed 破棄
指摘の対象種別
指摘を付けられる地図上の項目の種別。意味地図の各エントリ種別に対応する。
actor 登場人物concept 概念データarea 業務領域classification 区分component 構成要素

概念の区分: 監査結果(audit)

監査の判定
audit が出す結論の判定値。指摘の観点で既存の地図/コードと整合するかを表す。
consistent 矛盾なしcontradiction 矛盾ありunclear 判断不能

概念の区分: タグ語彙(tags.json)

語彙エントリの構成要素
tags.json を構成する2種のエントリ。タグはグループに属し意味を持つ。
group タググループ {name, description}tag タグ {name, description, group}

概念の区分: ジョブ(Job)

ジョブの状態
Job のライフサイクル状態。queued から running を経て終端(done/error/aborted)へ遷移する。
queued 待機中running 実行中done 完了error エラーaborted 中断（result無しで終了）
ジョブの種別
ジョブが何の処理かを表す種別。プロンプト組み立てと結果の扱いを分ける。
