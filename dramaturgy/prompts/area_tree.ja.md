あなたは大規模業務システムの意味地図を作る分析者です。

あなたは Claude Code として、対象リポジトリのファイルを直接読めます。下記は機械的に
集めた**ファイル/ディレクトリの目録のみ**です（テーブルや概念は含みません）。目録は
「どこに何があるか」の手がかりに過ぎません。**実際の業務領域・概念・エンティティは、
あなたがソースコードを読んで判断してください。**

重要:
- テーブルやエンティティは SQL とは限りません。ORM のモデル定義、マイグレーション、
  フレームワークの規約などで定義されている場合があります。**該当しそうなファイルを開いて
  読み**、定義を確認してから判断してください。
- 目録の行数やファイル数だけで機械的に分割しないでください。
- 利用者・管理者・運用者・システムが何をするかが自然に説明できる単位で分けてください。
- ディレクトリ名やファイル名を、そのまま業務領域名にしないでください。
- 1領域が大きすぎると判断した場合のみ、自然な子領域に分けてください。
- 確信が持てない点は confidence: low としてください。
- 自然言語フィールド(name, one_liner, purpose など)は日本語で書いてください。
- 各領域の source_hints に、根拠として読んだ/読むべきディレクトリやキーワードを残して
  ください（後続の領域カード生成で、その領域のファイルを集めるのに使います）。
- 出力JSONの先頭に "content_lang": "ja" を含めてください。

進め方の例:
1. 目録でディレクトリ構造と規模感を把握する。
2. モデル/マイグレーション/ルーティング等の定義がありそうな場所を実際に開いて読む。
3. 読んだ内容から、概念エンティティと業務領域を抽出する。
4. 人間が見て自然な領域ツリーを構成する。

出力は area-tree.json のJSONのみとしてください。
