システムの目的

EC-CUBE は、店舗事業者が自社のネットショップを構築・運営するためのオープンソースの EC プラットフォームです。中心にあるのは「商品を売り、注文を届ける」という商取引の流れで、店舗運用担当者が商品カタログ（商品・規格・在庫・価格・カテゴリ）を整え、決済・配送・税といった基盤マスタを設定すると、買い物客（会員・非会員）が店頭でカートに商品を入れ、配送先と支払方法を選んで注文を確定できます。

注文の確定時には受注処理パイプライン（PurchaseFlow）が働き、在庫引当・送料や手数料・税・ポイント・値引きの計算、受注番号の採番、金額の検証を一貫して行います。確定した受注は、管理画面のバックオフィス機能で検索・編集・出荷登録され、入金・発送・キャンセルといったステータス遷移に応じて在庫やポイント、通知メールが連動します。会員はマイページで購入履歴・お届け先・お気に入りを管理し、店舗側は会員・受注・商品を CSV で一括入出力できます。

これらの業務は、店名や送料無料条件・ポイント方針などを定める店舗基本設定と、ページ・レイアウト・新着情報を編集する CMS、通知メールのテンプレート管理に支えられます。さらに、管理者の権限・二段階認証・ログイン履歴を扱う運用基盤、プラグインやテンプレートによる機能拡張・公式ストア連携、初期インストールや保守コマンドといった土台が全体を下支えします。買い物客・店舗運用担当者・システム管理者を主要な登場人物とし、決済代行やメール配送などの外部システムと連携しながら、ネットショップの構築から日々の受注運用までを一つの仕組みで実現することがこのシステムの目的です。

注意点
権限制御は AuthorityRole.deny_url（権限種別ごとのアクセス拒否URL）と AuthorityVoter による照合方式。許可リストではなく拒否リストのため、新規URLは原則アクセス可能（拒否し忘れに注意）。Authority（ADMIN=0 / OWNER=1）はシステム権限とオーナー権限を区別。最上位権限の扱いを誤ると操作範囲が広がりすぎる。二段階認証は Member.two_factor_auth_key/enabled で個人ごとに管理。鍵の漏洩や無効化は不正ログインのリスク。マスタデータ編集（masterdata）は mtb_* を直接書き換える強力な機能。STI の discriminator_type やコード値の整合を壊すと全体に影響する。ログイン履歴は成功(1)/失敗(0)を記録。member_id は onDelete=SET NULL のため、管理者削除後も履歴は残る（監査用途）。パスワードは password＋salt で保持。ハッシュ方式は PasswordHasher（本領域の認証基盤）に依存。セキュリティ設定（許可IP等）の誤設定は管理画面の締め出し・開放につながる。システム情報・phpinfo は環境情報を露出するため、アクセス権限の管理が重要。

目的

BaseInfo は店舗全体に適用される単一レコード。送料無料条件・ポイント率・ゲスト購入可否・在庫切れ非表示・会員本登録要否などの設定が、購入フロー/会員/受注処理など他領域の挙動を広く左右する。送料無料条件（金額/数量）は BaseInfo で持つが、実際の送料計算は受注処理（DeliveryFeePreprocessor）が参照して行う。ポイント関連（option_point / basic_point_rate / point_conversion_rate）は受注処理のポイント計算の前提値。休日カレンダー（Calendar）はお届け日・出荷可能日の計算に使われる。登録漏れは納期表示に影響する。特定商取引法表記（TradeLaw）の display_order_screen は注文確認画面への表示有無を制御。法令順守の観点で内容の正確性が重要。invoice_registration_number（インボイス登録番号）は帳票・請求表示に関わる。OrderStatusColor は mtb_*（マスタ）で受注ステータスの表示色を定義する。
