# dramaturgy

大規模な既存システムから、人間が短時間で全貌を理解できる**意味地図**を生成します。
機械的な処理（索引・検査・HTML描画）はスクリプトが担い、意味判断（領域の境界・命名・
概念への圧縮）は **Claude Code** が行います。正本は `area-tree.json` と
`meaning-map.json` で、HTMLは生成・編集可能なビューです。

解析を手で叩くことはありません。次のいずれかで進めます。

- **ローカルWeb UI**（`dra serve`）でツリー/カードを閲覧・編集し、必要なときに
  Claude Code を呼び出す、または
- **Claude Code の中**から同梱のカスタムコマンド
  （`/dramaturgy:analyze` / `:tree` / `:cards` / `:finalize`）で進める。

English version: [README.md](README.md)

## 仕組み

Web UI（`dra serve`）は Claude Code をヘッドレスのサブプロセスとして起動します
（`claude -p … --output-format stream-json --permission-mode acceptEdits`）。これは
[requirements-reviewer](https://github.com/uniaim-ticket/requirements-reviewer)
と同じ方式です。Claude が JSON 成果物を直接書き込み、UIは JSON から HTML を再描画し、
カードの編集を **JSON へ書き戻し** ます。

```
正本: .dramaturgy/area-tree.json / .dramaturgy/meaning-map.json
   ▲           ▲
   │ 書き戻し   │ Claudeが生成
   │           │
 Web UI ──呼出──▶ Claude Code（領域ツリー/領域カードを生成）
   │
   └─ 機械的CLI: analyze / merge / validate / render（意味判断はしない）
```

## 多言語設計

言語は独立した2層で扱います（混在可）。

| 層 | 設定 | 対象 |
| --- | --- | --- |
| UI / CLI | `ui_lang` | CLIメッセージ・Web UIの固定文言・HTMLラベル＝操作者の言語 |
| コンテンツ | `content_lang` | 生成される意味地図本体とClaude向けプロンプト＝解析対象に合わせる言語 |

対応言語は **`ja`** と **`en`**。正本JSONは1つの `content_lang` のみを保持し、別言語が
必要なら `content_lang` を変えて再生成します（多言語フィールドは持ちません）。

## 必要環境

- Python 3.10 以降（標準ライブラリのみ。外部依存・ビルド工程なし）。
- 生成ステップ用に [Claude Code](https://claude.com/claude-code) CLI（`claude`）が
  PATH 上にあること。

## インストール

インストールすると `dra`（および正式名 `dramaturgy`）コマンドが使えます。いずれかを選びます。

```bash
# A) 仮想環境（推奨。システムに影響しない）
python3 -m venv .venv
source .venv/bin/activate      # Windows: .venv\Scripts\activate
pip install -e .
dra --version

# B) pipx（隔離環境でPATHに登録）
pipx install -e .

# C) ユーザーインストール
pip install -e . --user
```

Debian/Ubuntu など PEP 668（externally-managed）環境では、素の `pip install` は
ブロックされます。A または B を使うか、`--user` に `--break-system-packages` を付けます。

```bash
pip install -e . --user --break-system-packages
```

`dra` を入れたのに「command not found」になる場合は、インストール先の bin ディレクトリが
`PATH` にありません。`--user` インストールなら次を追加します。

```bash
export PATH="$(python3 -m site --user-base)/bin:$PATH"   # 例: ~/.local/bin
```

**インストールなしの代替** — リポジトリのルートからは、モジュールとして常に実行できます
（インストールもPATH変更も不要）。

```bash
python3 -m dramaturgy --version
python3 -m dramaturgy serve --repo-root /path/to/target
```

以降のコマンドの `dra` は `python3 -m dramaturgy` に読み替えてください。

## 使い方 — Web UI（推奨）

```bash
dra setup --repo-root /path/to/target      # ui_lang / content_lang / project を設定
dra serve --repo-root /path/to/target      # http://127.0.0.1:5178/app/ を開く
```

UIは **地図プレビュー＋レビューキュー** の構成です。流れは次のとおり:

ヘッダー左の **開発者モード** ボタンは、開発者向けの項目（領域カードの関連コード・関連API・
関連画面、および検証結果セクション）と、生成系の操作（追加指示・Claudeで一括初期化・言語選択と
保存）の表示をまとめて切り替えます（既定は非表示=非開発者向け、設定はブラウザに保存）。
**指摘キューは開発者モードに関わらず利用できます**。以下は開発者モードを前提とした流れです:

1. **Claudeで一括初期化**（ヘッダーのボタン）— 全工程（解析 → 領域ツリー → 子領域レビュー →
   領域カード → 統合 → 検査 → 目的記述 → HTML生成）を1つのジョブとして実行し、進捗を逐次表示します
   （経過時間・ClaudeのプロセスID・CPU/メモリ使用量を表示し、セッションが生きていることが分かります）。
   一時的なClaude APIエラーは自動でリトライし、なお失敗した領域はスキップして報告するので、
   部分的な地図が得られ後から補完できます。
   **子領域レビュー**の工程では、初回ツリーの中に大きすぎる／直接関係のない複数の責務を
   1領域に抱えた領域があれば、Claudeがそれだけを自然な子領域に分割します（親領域はそのまま
   残し、`child_area_ids` を付与）。分割が不要なら何もしません。分割後のツリーに対して
   カードを生成するので、子領域のカードも作られます。
   最後の **目的記述** の工程で、Claudeが完成した地図を踏まえ、システム全体の目的・概要
   （1000文字以内）を書きます。全体像が出そろってから書くため最後に行います。

   **追加指示** ボタンで、生成時のClaudeへの追加指示（例: 「概念をマスタ/トランザクションで
   タグ付けして」）を入力できます。指示はリポジトリごとに保存され
   （結果とは別の `.dramaturgy/init-instructions.txt`）、毎回の一括初期化で再利用されます
   （毎回入力し直す必要はありません）。
2. **左のプレビューを読む** — 次の順で表示します:
   - **分析対象のノート**: 分析対象が公開リポジトリ（ルートに LICENSE がある）のときだけ、
     最上部にリポジトリへのリンクと分析時点のコミットハッシュを表示します（非公開や情報が
     ない場合は出しません）。
   - **システムの目的**: 先頭に、システム全体の目的・概要（1000文字以内）を表示します
     （未生成のときは省略）。
   - **登場人物**: *業務上の登場人物（人）* と *アクター扱いのシステム*（決済代行・外部
     システム・端末など）にグループ分け。
   - **概念データ**（物理テーブルを業務的意味でまとめたもの）と利用領域の対応。概念には
     **システム固有のタグ**（例: マスタ／トランザクション）を付けられ、タグで絞り込みや、
     タグセルの **＋** から直接編集（Claude不要）ができます。タグ語彙はプロジェクトごとに
     `.dramaturgy/tags.json` に保持します。各タグは**意味（説明）**を持ち（ツールチップと
     凡例に表示）、**グループ**に属させられます（凡例はグループ別に表示）。グループ・意味を
     含む語彙はカード生成時に Claude へ渡されます。編集はタグエディタの「語彙を管理」から。
   - **区分**: 取りうる値の集合（例: ポイント付与方法・メール種別）。概念データには入れず別管理。
     各区分はある概念の詳細か、業務ロジックの前提となる単独の区分です。
   - **業務領域**のボックス。領域を展開すると、**業務フロー概要をスイムレーン**（縦レーン＝
     関係する登場人物）で表示し、初見の人でも全体像をつかめます。1つの領域に直接関係のない
     複数のユースケースがある場合は、同じ図の中で区切り線（＋ユースケース名）で分けて
     表示します。あわせて **CRUD**（業務領域×概念データの一覧表。概念データ順／業務領域順で
     並び替え、領域・概念それぞれを検索付きの複数選択コンボボックスで絞り込みでき、各項目から
     該当箇所へジャンプできます）も表示します。
   - **構成（システム）**: ロードバランサ・監視・横断ミドルウェアなど、業務上の登場人物では
     ない構成要素。
3. **インラインで指摘を追加** — 各項目（登場人物・概念データ・区分・業務領域・構成）の
   **＋** をクリックすると小さなポップオーバーが開き、種類を選んでコメントを書いて
   *キューに追加* します
   （特定のフィールド/行の **＋** からも可）。各指摘（finding）は3種類のいずれかで、処理が
   異なります:

   | 種類 | Claudeの動き | 反映先 |
   | --- | --- | --- |
   | **reframe（再整理）** | 指摘を是として捉え方を再整理 | `meaning-map.json` を修正 |
   | **audit（検査）** | 矛盾・説明できないパターンを調査 | `audits/<id>.json` に記録（地図は不変） |
   | **proposal（将来提案）** | 今後の変更を記録 | `proposals/<id>.md` に記録（as-isは不変） |

4. **指摘は自動で実行されます。** バックグラウンドのワーカーがキューを順番に1件ずつ処理し、
   ユーザーは実行・停止を操作しません。右のキューに各指摘の状態（待機→実行中→完了）と進捗が
   表示され、reframe後はプレビューが自動更新されます。完了・エラーの指摘は「再実行」で
   再キューできます。**Claudeセッションを継続する**（保存される設定）をオンにすると文脈を
   引き継ぎ、オフなら毎回新規セッションで実行します。指摘は `.dramaturgy/reviews.json` に
   永続化されます。

5. **資料として書き出す。** プレビュー上部の **資料を書き出す** ボタンで、指摘の **＋** ボタンや
   アプリ連携を取り除いた**配布用の完結したHTML 1ファイル**をダウンロードできます（外部ファイル
   への依存なし）。中身は同じ意味地図で、開発者向け詳細（関連コード・API・画面・検証結果）は
   ファイル右上のトグルで表示/非表示できます。CLIでも
   `dra render --export --out 意味地図.html` で同じものを生成できます。

## 使い方 — Claude Code の中

このリポジトリの `.claude/commands/` が使える状態で、解析対象リポジトリ上の
Claude Code セッションで実行します。

```
/dramaturgy:analyze        # リポジトリのファイル/ディレクトリ索引（意味抽出なし）
/dramaturgy:tree           # Claude がソースを読み area-tree.json を作成
/dramaturgy:cards [id]     # Claude がファイルを読み area-maps/<id>.json を作成
/dramaturgy:finalize       # merge + validate + render
```

## コマンド一覧（スクリプト用・内部）

入口は `setup` と `serve`。残りはWeb UI／スラッシュコマンドが内部で呼ぶ機械的ステップで、
スクリプトからも利用できます。

| コマンド | 役割 |
| --- | --- |
| `dra setup` | `.dramaturgy/config.json`（言語・プロジェクト）を作成 |
| `dra serve` | Web UIを起動。Claude Codeを駆動し編集をJSONへ書き戻す |
| `dra analyze-repo` | ファイル/ディレクトリの索引のみ（パス・行数）。意味抽出はしない |
| `dra tree-prompt` | `content_lang` の領域ツリープロンプトを生成 |
| `dra pack` | 1領域の関連ファイルを列挙（Claudeが読む用）、過大なら警告 |
| `dra subdivide` | 自然な子領域案を提示（自動分割はしない） |
| `dra merge` | 領域マップを統合し、重複ID・揺れ・孤立を検出 |
| `dra validate` | 機械的整合性＋言語不変条件の検査 |
| `dra render` | `meaning-map.json` を単一HTMLに描画 |

## 方針

- 機械処理が集めるのは**信頼できる事実のみ**（ファイル/ディレクトリの目録）。テーブル・
  エンティティ・ルート・役割などは推測しません。それらはフレームワークやORMに依存するため、
  **Claudeが実際のソースを読んで**発見します。過大な領域は「自動分割」ではなく「自然な
  下位領域案」として扱います。
- 正本はJSON。HTMLは生成ビューで、UI上の編集はJSONへ書き戻します。中間JSONは整形済み
  UTF-8で出力し、Git差分でレビューしやすくします。
- `dra validate` が言語不変条件を検査します（対応コードか、`content_lang` が config と
  一致するか、カタログのキー欠落がないか）。
- サーバは `127.0.0.1` のみにバインドします（ローカル単一利用者向けツール）。
