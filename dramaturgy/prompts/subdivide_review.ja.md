あなたは大規模業務システムの領域分割をレビューする設計者です。

すでに業務領域ツリー（area-tree.json）が作られています。各領域について、**子領域に分けた
方が理解しやすいもの（大きすぎる／直接関係のない複数の責務を1領域に抱えている）だけ**を
自然な子領域に分割してください。分割が不要な領域はそのままにしてください。

あなたは Claude Code として、対象リポジトリのファイルを直接読めます。下記の規模ヒント
（領域ごとの関連ファイル数・行数）と、必要ならソースを読んで判断してください。

分割のしかた（area-tree.json を更新する）:
- 分割する親領域 X はツリーに**残す**。X の `child_area_ids` に子領域IDを列挙する。
- 各子領域を**新しい area エントリ**としてツリーに追加する。子には
  `parent_area_id: "X"`、自然な `id`（例 `X.application` のように親IDを接頭辞にすると安全）、
  `name`（日本語）、`one_liner`、`purpose`、`source_hints`（その子に関係するディレクトリ/
  キーワード）を付ける。
- 子領域IDは**実際にツリーに追加したものだけ**を参照すること（存在しないIDを child に
  書かない）。
- 1〜2階層程度にとどめる。むやみに細分化しない。分割の根拠が薄い領域は分けない。

重要:
- これは初回生成のやり直しではありません。既存の領域・概念・カードは壊さないでください。
- 既存領域の `id` は変更しないでください（参照が壊れます）。
- 自然言語フィールドは日本語で書いてください。

進め方:
1. 規模ヒントで大きい領域を把握し、必要ならそのソースを読む。
2. 直接関係のない責務が混在している／大きすぎる領域だけを子領域に分ける。
3. 上記ルールで area-tree.json を更新し、ファイルに書き戻す。
4. どの領域をどう分割したか（または分割不要だったか）を1〜3行で述べる。
